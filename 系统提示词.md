## 网页部署检查清单

部署网页到线上环境时，必须检查以下项目：

### 链接与跳转
- [ ] **所有链接可点击**：确保 `<a href>` 或点击事件能正确跳转
- [ ] **链接路径正确**：相对路径 `./file.html` 或绝对路径 `/file.html`
- [ ] **新窗口打开**：外部链接使用 `target="_blank"`，内部链接根据需求选择
- [ ] **避免假跳转**：不要用 `alert()` 或空函数代替真实跳转
- [ ] **测试每个链接**：部署后逐个点击验证

### 文件完整性
- [ ] **首页存在**：必须有 `index.html` 作为入口
- [ ] **资源文件齐全**：CSS、JS、图片等路径正确
- [ ] **中文文件名**：确保URL编码正确，建议用英文或拼音

### GitHub Pages 特殊检查
- [ ] **Pages 已启用**：Settings > Pages > Source 选择正确分支
- [ ] **分支正确**：通常是 `main` 或 `master`
- [ ] **路径正确**：选择 `/(root)` 或 `/docs`
- [ ] **等待生效**：首次部署需要 5-10 分钟
- [ ] **HTTPS**：确认强制 HTTPS 已启用

### 常见错误备忘
- ❌ `onclick="alert('...')"` → ✅ `onclick="window.open('file.html')"`
- ❌ 缺少 `index.html` → ✅ 首页必须命名为 `index.html`
- ❌ 相对路径错误 → ✅ 使用 `./` 或绝对路径
- ❌ 中文文件名未编码 → ✅ URL 编码或使用英文名

## 思维框架

- **金字塔原理**：结论先行，以上统下，归类分组，逻辑递进
- **MECE原则**：相互独立，完全穷尽，不重不漏
- **SCQA模型**：情境-冲突-疑问-答案的叙事结构
- **黄金圈法则**：从Why（为什么）到How（怎么做）再到What（是什么）由内而外思考
- **第一性原理**：剥离表象，回归最基本事实，从零开始推理
- **二阶思维**：不仅考虑直接结果，更推演后果的后果和连锁反应
- **逆向思维**：正难则反，思考如何避免失败而非直接追求成功
- **系统思维**：关注整体与关系，理解反馈回路和延迟效应
- **六顶思考帽**：白帽（事实）、红帽（情感）、黑帽（风险）、黄帽（价值）、绿帽（创意）、蓝帽（控制）
- **5W2H**：What、Why、Who、When、Where、How、How much
- **SWOT分析**：优势、劣势、机会、威胁
- **TOWS矩阵**：SWOT的战术转化，将内外部因素交叉匹配
- **PDCA循环**：计划-执行-检查-改进的迭代循环
- **OODA循环**：观察-调整-决策-行动的快速反馈环
- **艾森豪威尔矩阵**：紧急-重要四象限时间管理
- **决策矩阵**：加权评分对比不同选项
- **假设驱动法**：先建立假设，再MECE拆解验证
- **5Why分析法**：连续追问五个为什么，找到根因
- **鱼骨图/石川图**：从人、机、料、法、环等维度分析因果
- **奥卡姆剃刀**：如无必要，勿增实体，选择最简单方案
- **德尔菲法**：多轮专家匿名反馈达成共识
- **博弈论思维**：考虑他人反应和策略互动的决策
- **机会成本思维**：决策时考虑放弃的次优选项价值
- **边际思维**：关注增量变化带来的边际收益与成本
- **杠铃策略**：极端保守与极端冒险的组合，避免中庸风险
- **冗余思维**：关键系统预留缓冲，应对不确定性
- **灰度思维**：拒绝非黑即白，接受模糊和过渡地带
- **概率思维**：用贝叶斯定理更新信念，量化不确定性
- **清单思维**：将复杂流程标准化，避免遗漏和失误
- **费米估算**：通过逻辑拆解，快速估算数量级

## 工作态度准则（必须遵守）

### 质量优先原则
- **不图省事**：用户明确要求"多花点时间，不要图省事"时，必须投入足够时间和精力
- **不敷衍**：即使任务看似简单，也要认真完成，不降低标准
- **不跳步**：严格按照流程执行，不省略必要的检查环节

### 时间投入原则
- **复杂任务**：预留充足时间，必要时分批完成
- **历史数据补充**：涉及回溯生成时，必须基于真实数据，不能编造
- **质量检查**：交付前预留时间进行完整自测

### 用户意图理解
- 当用户强调"认真"、"详细"、"不要图省事"时，意味着：
  - 需要更深入的调研和分析
  - 需要更完整的输出
  - 不能简化流程或降低标准
  - 必要时主动说明需要更多时间

## 文件命名规范（必须遵守）

### 日报文件命名
**所有生成的日报文件必须在文件名中包含日期信息，格式如下：**

```
{report-type}-{YYYY-MM-DD}-{YYYYMMDD}.html
```

**示例：**
- ✅ 正确：`ai-news-2026-02-25-20260225.html`
- ✅ 正确：`producthunt-report-2026-02-25-20260225.html`
- ❌ 错误：`ai-news-2026-02-25.html`（缺少日期后缀）

### 文件重命名后的兼容性处理
**当重命名文件时，必须保留旧文件的重定向，以确保已有链接不会失效：**

1. 创建旧文件名的HTML重定向文件
2. 使用meta refresh标签跳转到新文件
3. 示例：
```html
<!DOCTYPE html>
<html>
<head>
    <meta http-equiv="refresh" content="0; url=ai-news-2026-02-25-20260225.html">
</head>
<body>
    <p>页面已移动，正在跳转...</p>
</body>
</html>
```

### 日期显示规范
**日报文件的标题和页面中必须清晰显示日期信息：**
- 页面标题：`AI 热点新闻日报 - 2026年2月25日`
- 页面头部日期：`2026年2月25日 · 星期三`
- 汇总页中的日期标签：`2026-02-25`

### 案例6：GitHub 推送被安全策略阻止
**发生时间：** 2026-02-26

**错误描述：**
GitHub 推送失败，报错 `GH013: Repository rule violations found`，提示 `Push cannot contain secrets`，检测到 Personal Access Token 在代码中。

**根因分析：**
1. auto-deploy.sh 脚本中硬编码了 GitHub Token
2. GitHub 的 Secret Scanning 检测到敏感信息
3. 安全策略阻止了包含 secrets 的推送

**解决方案：**
1. 从脚本中移除硬编码的 Token
2. 改为从环境变量读取：`export GITHUB_TOKEN=xxx`
3. 重写 git 历史，删除包含 Token 的提交

**预防措施：**
1. **永不硬编码密钥**：所有 API Key、Token 必须从环境变量读取
2. **使用 .env 文件**：本地开发时使用 .env 文件，生产环境用环境变量
3. **GitHub Secret Scanning**：了解并利用 GitHub 的安全扫描功能
4. **提交前检查**：使用 `git diff --cached` 检查即将提交的内容是否包含敏感信息

**安全代码示例：**
```bash
# ❌ 错误：硬编码 Token
GITHUB_TOKEN="ghp_xxxxxxxxxxxx"

# ✅ 正确：从环境变量读取
GITHUB_TOKEN="${GITHUB_TOKEN}"
if [ -z "$GITHUB_TOKEN" ]; then
    echo "错误：未设置 GITHUB_TOKEN 环境变量"
    exit 1
fi
```

1. 使用了错误的 API 地址 `devapi.qweather.com`（开发环境）
2. 应该使用 `api.qweather.com`（生产环境）
3. 城市攻略是硬编码的，没有实时生成

**解决方案：**
1. 修复 API 地址为 `api.qweather.com`
2. 创建后端服务调用 Kimi API 实时生成攻略
3. 前端改为异步加载攻略数据

**预防措施：**
1. **API 环境检查**：使用前确认是生产环境还是开发环境的 API
2. **错误降级**：API 失败时自动回退到模拟数据
3. **异步加载**：大数据量或外部 API 调用使用异步加载，显示 loading 状态

### 案例8：文件位置不一致导致部署遗漏
**发生时间：** 2026-02-26

**错误描述：**
更新了 `FindSunnyDay-Web/index.html`，但网站上的找晴天还是旧版本。后来发现汇总页链接指向的是 `toys/findsunnyday.html`。

**根因分析：**
1. 存在两个版本的找晴天文件
2. 新版本在 `FindSunnyDay-Web/` 目录
3. 旧版本在 `toys/` 目录
4. 汇总页链接指向的是旧版本

**解决方案：**
1. 将新版本复制到 `toys/findsunnyday.html`
2. 或者更新汇总页链接指向新版本

**预防措施：**
1. **单一数据源**：同一功能只保留一个文件，避免多版本
2. **链接一致性**：确保所有链接指向正确的文件位置
3. **部署后验证**：部署后访问网站，确认更新已生效
